7. Workflow Execution Engine

Execution Flow

1. User clicks "Execute" in frontend
2. Frontend sends POST /api/workflows/{id}/execute
3. Backend API validates workflow + enqueues Celery task
4. Returns execution_id immediately (non-blocking)
5. Celery worker picks up task from queue
6. WorkflowExecutor initializes:
   - Loads workflow definition
   - Builds DAG (Directed Acyclic Graph)
   - Performs topological sort for execution order
7. Execute nodes sequentially/parallel based on dependencies
8. Each node:
   - Gets previous node data as input
   - Validates configuration
   - Executes with retry logic (exponential backoff)
   - Returns result + logs
   - Updates execution state in Cosmos DB
9. Send real-time updates via WebSocket to frontend
10. On completion, save final result to database

State Machine

Status Description Next States
PENDING Workflow queued, not started RUNNING, CANCELLED
RUNNING Workflow executing SUCCESS, FAILED, CANCELLED
SUCCESS Completed successfully -
FAILED Error occurred during execution -
CANCELLED User cancelled execution -

Error Handling & Retries

  • Node-level retries: Exponential backoff (2^attempt seconds)
  • Max retries per node: 3 (configurable)
  • Continue on fail: Optional node setting to continue workflow
  • Dead-letter queue: Failed executions moved to DLQ for analysis
  • Circuit breaker: Stop calling failing external APIs temporarily
